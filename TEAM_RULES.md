# TEAM_RULES

Правила командної роботи над проєктом (HTML + CSS, Vite, GitHub).

---

## 1. Git / GitHub Flow

### 1.1. Головна гілка

- Головна гілка: **`main`**.
- У `main` лежить **тільки стабільний і перевірений код**.
- Прямі коміти в `main` **заборонені** – всі зміни йдуть через Pull Request (PR).

### 1.2. Робочі гілки

- Для кожної задачі/секції створюється окрема гілка **від `main`**.
- Формат назви гілки:

  ```bash
  feature/header
  feature/about
  feature/footer
````

Одна секція = одна гілка (включно з адаптивом).

* Створення гілки:

  ```bash
  git switch -c feature/header
  ```

### 1.3. Оновлення гілки перед пушем

Перед тим як пушити або створювати PR, онови свою гілку від актуального `main`:

```bash
git checkout main
git pull origin main
git checkout feature/header
git merge main
```

Якщо є конфлікти – вирішуємо локально, перевіряємо проєкт, комітимо й пушимо.

### 1.4. Pull Request

* Назва PR:

  ```text
  feature: секція about (mobile, tablet, desktop)
  ```

* В описі PR вказати:

  * що зроблено;
  * стан адаптиву (що покрито: mobile / tablet / desktop);
  * що залишилося доробити (якщо є);
  * бажано додати скріншоти (мінімум mobile + desktop).

* PR **мерджиться тільки після рев’ю** від колеги.

* Не мерджимо самі себе без погодження (мінімум 1 approve).

## 2. Структура проєкту

Базова структура:

```text
src/
  partials/
    header.html
    about.html
    footer.html
  css/
    base.css
    header.css
    about.css
    footer.css
  img/
  main.js
  main.css
```

### 2.1. HTML

* Розмітка секцій – у `src/partials/`:

  * `header.html`, `about.html`, `footer.html` тощо.
* Головний шаблон (index.html або main layout) підключає partial-файли.

### 2.2. CSS

* Стилі секцій – у `src/css/`:

  * `header.css`, `about.css`, `footer.css` – **один файл на секцію**.
* Загальні стилі – тільки в `base.css`:

  * reset / normalize;
  * шрифти;
  * кольори;
  * CSS-змінні (`:root { --color-primary: ... }`);
  * спільні утиліти (`.container`, базові кнопки тощо).
* Повторювані стилі **не дублюємо** в секційних файлах – виносимо в `base.css`.

---

## 3. HTML-правила

### 3.1. Семантика

Використовуємо семантичні теги:

* `header`, `main`, `section`, `footer`, `article`, `nav`, `ul`, `ol`, `li` тощо.
* Один `<h1>` на сторінку.
* У секціях – `<h2>` як заголовок секції, далі `<h3>` і т.д. за ієрархією.

### 3.2. Класи та іменування (BEM-підхід)

* Для кожної секції – **свій блок**:

  ```html
  <section class="about">
    <div class="container">
      <h2 class="about__title">Про нас</h2>
      <p class="about__text">...</p>
    </div>
  </section>
  ```

* Префікс класу завжди починається з назви секції:

  ```text
  about__title
  about__text
  header__nav
  header__logo
  footer__link
  ```

* Модифікатори:

  ```text
  about__title--accent
  header__btn--primary
  ```

* Клас `.container`:

  * використовується тільки для **вирівнювання та обмеження ширини**;
  * має один спільний стиль у `base.css`.

### 3.3. Атрибути

Рекомендований порядок атрибутів:

```html
<img
  class="about__image"
  id="team-photo"
  src="img/team.webp"
  alt="Командне фото"
  title="Наша команда"
/>
```

* Використовуємо тільки **подвійні лапки** `" "`.

### 3.4. Доступність (a11y)

* У всіх `img` має бути осмислений `alt`, окрім чисто декоративних зображень (`alt=""`).
* Не видаляємо `outline` у фокусних елементів (`button`, `a`, `input`) без заміни на власний фокус-стиль.
* Текст у кнопках та лінках має бути зрозумілим («Докладніше», «Замовити», «Дізнатися більше»).

## 4. CSS-правила

### 4.1. Іменування класів

* Імена класів починаємо з назви секції: `header__`, `about__`, `footer__` тощо.
* Не використовуємо “універсальні” класи типу `.title`, `.btn`, `.text` без префікса блоку.
* Для повторюваних елементів (типу кнопок) – або:

  * окремий утилітарний клас у `base.css` (`.btn-primary`),
  * або спільний патерн + модифікатори.

### 4.2. Mobile-first

Пишемо стилі в mobile-first підході:

```css
/* мобілка */
.about {
  /* базові стилі */
}

@media (min-width: 768px) {
  /* планшет */
  .about {
    /* адаптація */
  }
}

@media (min-width: 1440px) {
  /* десктоп */
  .about {
    /* десктоп-адаптація */
  }
}
```

Точки перелому (за замовчуванням):

* `min-width: 768px` – планшет;
* `min-width: 1440px` – десктоп.

(Мобільна версія – базова, без медіа-запитів.)

### 4.3. Сітка, одиниці, вкладеність

* Сітка – тільки **flex** (CSS Grid у цьому проєкті не використовуємо).
* Одиниці вимірювання: `px`, `%`.
* Селектори для стилізації – **тільки класи** (без прив’язки до тегів і id).
* Максимальна вкладеність селекторів: **2 рівні**:

  ```css
  .about__list {
    ...
  }

  .about__list-item {
    ...
  }

  /* допустимо: */
  .about__list-item .about__icon {
    ...
  }
  ```

### 4.4. `!important`

* **Не використовуємо `!important`**, окрім крайніх випадків.
* Якщо без нього ніяк – додаємо коментар із поясненням:

  ```css
  .about__title {
    margin-bottom: 24px !important; /* тимчасово: перекриваємо сторонні стилі */
  }
  ```

### 4.5. base.css та змінні

У `base.css` зберігаємо:

```css
:root {
  --color-primary: #000000;
  --color-accent: #ff6b08;
  --color-bg: #ffffff;
  --font-main: 'Roboto', system-ui, sans-serif;
}

body {
  font-family: var(--font-main);
  color: var(--color-primary);
  background-color: var(--color-bg);
}
```

* У секційних CSS **не прописуємо довільні кольори напряму** – беремо тільки зі змінних в `:root`.

## 5. Зображення та ресурси

* Фото: `.webp`, `.jpg`.
* Іконки, логотипи: `.svg`.
* Не заливаємо в репозиторій зображення по 5–20 МБ.
* Розмір картинки повинен відповідати реальному використанню (не 4000px шириною для блоку 300px).
* Шляхи до зображень – відносні до `src/img`.

## 6. Коміти

### 6.1. Формат повідомлень

Формат:

```text
feat(header): додано HTML і стилі мобільної версії
style(header): адаптив для планшета
fix(header): виправлені відступи на мобілці
refactor(header): спрощено структуру розмітки
docs: оновлено TEAM_RULES.md
```

* Коміт має бути **коротким і змістовним**.
* Одна логічна зміна – один коміт.

### 6.2. Мова комітів

* Одна мова для всієї команди: **або українська, або англійська**.
* Команда домовляється один раз і дотримується вибору.

## 7. Робота з репозиторієм (для новачків)

### 7.1. Як завантажити проєкт

1. Встановити Node.js (версія LTS).

2. Клонувати репозиторій:

   ```bash
   git clone https://github.com/dmytro-dobrotvorskyi/project-08.git
   ```

3. Перейти в папку проєкту:

   ```bash
   cd project-08
   ```

4. Встановити залежності:

   ```bash
   npm install
   ```

### 7.2. Запуск проєкту локально

```bash
npm run dev
```

Сайт буде доступний за адресою, яку покаже консоль (зазвичай `http://localhost:5173/`).

### 7.3. Збірка проєкту

```bash
npm run build
```

Це створить оптимізовану продакшн-версію.

## 8. Робота з чужими змінами

### 8.1. Оновлення main

Якщо хтось оновив `main`, підтягнути зміни:

```bash
git checkout main
git pull origin main
```

### 8.2. Оновлення своєї гілки

```bash
git checkout feature/header
git merge main
```

* Якщо виникли конфлікти:

  * вирішуємо їх у редакторі;
  * запускаємо проєкт, перевіряємо, що все працює;
  * робимо коміт:

    ```bash
    git add .
    git commit -m "fix(header): resolve merge conflicts"
    git push origin feature/header
    ```

## 9. Що не пушимо в репозиторій

* Папки `node_modules/`, `dist/` – мають бути в `.gitignore`.
* Локальні файли IDE/редактора (якщо не домовлено навпаки).
* Тимчасові файли, особисті чернетки, тестові картинки.

---

## 10. Комунікація в команді

* Якщо щось ламається – **пишемо в командний чат**, не пушимо в `main` «якось працює».
* Якщо сумніваєтесь у назві класу або рішенні – запитуємо в команді.
* Не змінюємо чужу секцію без попередження:

  * пишемо автору в чат;
  * описуємо зміни в PR з позначкою автора (mention).
